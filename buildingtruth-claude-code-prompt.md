# BuildingTruth — Claude Code Project Prompt

## Project Overview

BuildingTruth is a public data-driven transparency platform that aggregates San Francisco public records to generate accountability scores for residential landlords. The site allows renters to look up a landlord before signing a lease and see their track record across DBI complaints, eviction filings, rent board petitions, 311 complaints, and permit investment — all sourced from the city's open data APIs.

This is a Next.js application deployed on Vercel. It uses a static-generation approach with a scheduled data pipeline that pulls from SF Open Data SODA APIs, runs a scoring engine, and outputs pre-computed JSON that the frontend renders. No user-generated content at launch. No authentication. No database server — just static JSON files generated by the pipeline.

## Phase 1: Data Pipeline

### 1.1 SODA API Ingestion

Create `/scripts/ingest.ts` — pulls raw data from these 5 SF Open Data endpoints:

| Dataset | Endpoint | Key Fields to Extract |
|---|---|---|
| DBI Complaints & Violations | `https://data.sfgov.org/resource/nbtm-fbw5.json?$limit=100000` | complaint_number, date_filed, block, lot, street_number, street_name, street_suffix, status, nov_category_description, nov_item_description |
| Eviction Notices | `https://data.sfgov.org/resource/5cei-gny5.json?$limit=100000` | eviction_id, address, city, file_date, non_payment, breach, nuisance, illegal_use, failure_to_sign_renewal, access_denial, unapproved_subtenant, owner_move_in, demolition, capital_improvement, substantial_rehab, ellis_act_withdrawal, condo_conversion, roommate_same_unit, other_cause, late_payments, lead_remediation, development, good_samaritan_ends |
| Rent Board Petitions | `https://data.sfgov.org/resource/6swy-cmkq.json?$limit=100000` | petition fields (extract all available — filing grounds, outcomes, dates, addresses) |
| 311 Cases | `https://data.sfgov.org/resource/vw6y-z8j6.json?$where=service_name='Building and Property Maintenance' OR service_name='Blocked Street or SideWalk' OR category='Street and Sidewalk Cleaning'&$limit=100000` | service_request_id, service_name, service_details, address, opened, closed, status |
| Building Permits | `https://data.sfgov.org/resource/i98e-djp9.json?$limit=100000` | permit_number, permit_type, permit_type_definition, status, status_date, filed_date, description, estimated_cost, street_number, street_name, street_suffix |

**Important API notes:**
- No API key required
- Default SODA limit is 1000 rows — ALWAYS append `$limit=100000` or paginate with `$offset`
- Use `$where` clauses to filter 311 data to housing-related categories
- Cache raw responses to `/data/raw/` as timestamped JSON files

### 1.2 Address Normalization

Create `/scripts/normalize.ts` — standardize all addresses to a canonical format:
- Uppercase

- Normalize street suffixes (ST, STREET → ST; AVE, AVENUE → AVE; etc.)
- Generate a composite key: `{street_number}-{street_name}-{street_suffix}`

## Phase 2: Scoring Engine

Create `/scripts/score.ts`

### Composite Score: 0–100 (higher = better landlord)

Each landlord receives a score calculated as follows:

#### Signal 1: DBI Violations (30% weight)
- Count open/unresolved complaints per unit
- Weight by severity: housing code violations > electrical > plumbing > other
- Recency decay: last 3 years = 1.0x, 3-5 years = 0.5x, 5+ years = 0.25x
- Raw metric: complaints per unit per year (lower is better)

#### Signal 2: Eviction Filing Rate (25% weight)
- Count total eviction notices per unit
- Apply 2x multiplier on no-fault evictions: `ellis_act_withdrawal`, `owner_move_in`, `demolition`, `condo_conversion`
- Recency decay same as above
- Raw metric: eviction notices per unit per year (lower is better)

#### Signal 3: Rent Board Petition Outcomes (20% weight)
- Count tenant petitions filed: decrease in services, illegal increase, failure to repair
- Weight sustained/granted findings at 2x vs. dismissed/withdrawn
- Raw metric: sustained tenant petitions per unit per year (lower is better)

#### Signal 4: 311 Complaint Density (15% weight)
- Count habitability-related 311 calls per unit
- Filter to relevant categories only (building maintenance, pest, mold, plumbing, heat, etc.)
- Recency decay same as above
- Raw metric: complaints per unit per year (lower is better)

#### Signal 5: Permit & Maintenance Investment (10% weight — POSITIVE signal)
- Count building permits for improvements/maintenance (exclude demolition permits)
- Higher permit activity = positive signal (landlord investing in property)
- Raw metric: permits per unit per year (higher is better)

#### Normalization
1. For each signal, calculate the raw per-unit-per-year metric for every landlord
2. Percentile rank each landlord across the full roster (0th = worst, 100th = best)
3. Invert negative signals (complaints, evictions, petitions, 311) so that fewer = higher score
4. Do NOT invert positive signal (permits) — more = higher score
5. Apply weights and sum to composite 0-100 score
6. Round to nearest integer

#### Output
Write to `/data/leaderboard.json`:
```json
{
  "generated_at": "2026-03-22T00:00:00Z",
  "landlords": [
    {
      "id": "veritas-investments",
      "name": "Veritas Investments",
      "slug": "veritas-investments",
      "composite_score": 34,
      "tier": 1,
      "estimated_units": 3000,
      "neighborhoods": ["Nob Hill", "Russian Hill"],
      "scores": {
        "dbi_violations": { "raw": 0.12, "percentile": 22, "weighted": 6.6 },
        "eviction_rate": { "raw": 0.08, "percentile": 18, "weighted": 4.5 },
        "rent_board_petitions": { "raw": 0.04, "percentile": 30, "weighted": 6.0 },
        "three_eleven_density": { "raw": 0.15, "percentile": 25, "weighted": 3.75 },
        "permit_investment": { "raw": 0.02, "percentile": 45, "weighted": 4.5 }
      },
      "total_dbi_complaints": 360,
      "total_eviction_notices": 240,
      "total_rent_board_petitions": 120,
      "total_311_cases": 450,
      "total_permits": 60,
      "address_coverage": "partial"
    }
  ],
  "top_3": [ /* 3 highest scoring landlords */ ],
  "bottom_3": [ /* 3 lowest scoring landlords */ ]
}
```

Write individual landlord detail files to `/data/landlords/[slug].json` containing full record-level data for the dashboard charts.

## Key Constraints & Guardrails

- **No user-generated content.** Data only comes from public APIs and the curated seed file.
- **No authentication.** Fully public, read-only site.
- **No database server.** All data is static JSON generated by the pipeline. This keeps hosting free and simple.
- **Transparent methodology.** Every score must be traceable back to source data. Link to DataSF datasets.
- **Legal framing.** The site presents a "Transparency Index," not verdicts. Include disclaimers on every dashboard. Use "Most Complaints" not "Worst Landlords" for the bottom-3 leaderboard.
- **Coverage honesty.** Always show whether a landlord's address portfolio is fully or partially mapped. Never present partial data as complete.
- **Mobile-first responsive design.** Many renters will access this on their phones while apartment hunting.
- **Accessibility.** WCAG 2.1 AA minimum. Score colors must have text labels, not just color coding.
- **Performance.** Static generation means pages should load fast. Target <2s LCP.
